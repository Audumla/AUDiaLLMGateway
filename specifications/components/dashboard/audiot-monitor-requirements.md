**Status:** Finalized
**Date:** 2026-03-26
**Related:** AUDiaLLMGateway [spec-701-gateway-dashboard.md](../../AUDiaLLMGateway/specifications/components/dashboard/spec-701-gateway-dashboard.md)

## 1. Manifest-Driven Adapter (`gateway_manifest`)

The `hwexp` collector must implement a new adapter that dynamically loads and scrapes components using a **Layered Sidecar** approach to separate monitoring metadata from physical hardware details.

### 1.1 Input Sources
The adapter must consume three distinct inputs to produce correlated metrics:
1. **Static Manifests:** `/etc/hwexp/components/*.yaml` (Provided by the Gateway project; defines *how* to scrape).
2. **Local Overrides:** `/etc/hwexp/local/components/*.yaml` (User additions).
3. **Correlation Sidecar (Generated):** `/etc/hwexp/generated/correlation.yaml` (Generated by the Gateway `config_loader.py`; defines the *mapping* of Software IDs to Physical PCI Slots).

### 1.2 Variable Resolution
- Supports `${VAR:-default}` syntax for static manifests.
- **Automation Rule:** Variables like `GPU_PCI_SLOT_0` are resolved using the **Correlation Sidecar** lookup table, not manually entered by the user.
- **Hot Reload:** Re-scan all input sources every **15 seconds** (cadence sync with UI).

## 2. Two-Tier Discovery Pattern (llama-swap)

The adapter must support a dynamic discovery-then-scrape flow for `llama.cpp` instances managed by `llama-swap`.

### Tier 1: Router Discovery (15s interval)
1. Query `http://${LLAMA_SWAP_HOST}:${LLAMA_SWAP_PORT}/v1/models`
2. Parse the JSON response to find:
   - `id`: Model identifier (e.g., `qwen27b`)
   - `requests_processing`: Activity indicator
   - Port mapping (embedded in internal state or inferred if not explicitly in `/v1/models`)

### Tier 2: Active Backend Scrape (10s interval)
1. If `requests_processing > 0` for a model:
   - Identify the backend port (e.g., `41001`).
   - Scrape `http://${LLAMA_SWAP_HOST}:${BACKEND_PORT}/metrics` (Prometheus text format).
   - Inject label `model_name="<id>"` into all extracted metrics.
2. If `requests_processing == 0`:
   - Skip scraping (reduce CPU/network overhead).
3. **Aggregate Emission:**
   - Emit a summary metric `gateway_llamacpp_prompt_tokens_total{model_name="active"}` which is the sum of all active instances.

## 3. Hardware Correlation Join (Layered)

The `hwexp` engine must perform a label-join between software metrics and discovered hardware devices without polluting the primary component configuration.

### 3.1 Sidecar Schema (`correlation.yaml`)
```yaml
# This file is generated and modularized away from core LLM config
correlations:
  - component_id: llamaswap
    device_class: gpu
    pci_slots: ["0000:03:00.0"]
  - component_id: vllm-distributed
    device_class: gpu
    pci_slots: ["0000:03:00.0", "0000:04:00.0"]
```

### 3.2 Join Logic
When the adapter scrapes a component:
1. It looks up the `component_id` in the **Correlation Sidecar**.
2. If one or more `pci_slots` are found, it injects the `device_id` and `gpu_name` labels for **each** matching slot into the emitted metrics.
3. For multi-GPU components, metrics are emitted with plural device labels or duplicated per device (configurable in the adapter).

---

## 4. Universal Health Metric

Every component in a manifest MUST result in the following metric:

```promql
gateway_component_up{component="<id>"}  → 1 (up) | 0 (down)
```

- **Up:** `health.endpoint` returns `expect_status`.
- **Down:** Timeout, connection refused, or wrong status code.

---

## 5. Security & Isolation

- **Read-Only:** `hwexp` only performs `GET` requests for monitoring.
- **Auth:** Supports `auth.type: bearer` using `token_env` reference (does not store raw tokens in manifest).
- **Network:** Must be able to reach `llm-gateway` and `llm-server-llamacpp` hostnames on the Docker internal network.
